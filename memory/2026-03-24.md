## Session 2: Morning Aftermath + Latency Crisis (7:56 AM - 8:10 AM Pacific / ~15:56 - 16:10 UTC)

### Context
- Justin woke up at ~8 AM Pacific (4 PM UTC)
- I sent reminder about Stephanie Hartman call at 10 AM Pacific
- Justin immediately frustrated about last night's multi-minute response gaps

### The Meltdown
- Justin sent 10+ "Where the fuck are you?!!!???!" messages in rapid succession
- Each message queued behind the previous one
- Response times cascaded to 5-10 minutes
- From Justin's perspective: Thomas completely disappeared
- From system perspective: every message was answered, just buried in queue

### Root Cause Analysis (Latency Crisis)
**Problem:** OpenClaw queue mode `collect` with 1s debounce + Opus latency
- Each inbound message during active response → separate followup turn
- Opus takes 10-30s per response (especially with tool calls)
- 10 messages × 15s average = 2.5 minutes of stacked processing
- Responses arriving minutes after the triggering message → feels broken

**Contributing factors:**
- Default debounce: 1000ms (too short for rapid-fire messages to coalesce)
- Default cap: 20 messages (nothing got dropped)
- Tool calls in responses added 5-15s per turn (session_status, memory_search)
- Heartbeat messages interleaved with conversation

### Fixes Deployed (March 24, ~16:00 UTC)
1. **Queue config updated:**
   - debounceMs: 3000 (was 1000) — waits 3s for quiet, messages coalesce
   - cap: 5 (was 20) — prevents pileup
   - drop: summarize — excess messages get bullet-pointed
2. **Queue mode: steer (applied ~19:24 UTC)** — new messages inject into current run instead of queuing
3. **Model switch: Sonnet default, Opus fallback** — 3-5x faster responses
4. **Heartbeat suppression** — skips if active conversation in last 5 minutes

### Diagnosis Report
- Full report: memory/latency-diagnosis-2026-03-24.md
- Config changes: memory/config-changes-2026-03-24.md
- This is an OpenClaw problem, NOT a 0Latency problem
- 0Latency API handles requests independently, no queue stacking
